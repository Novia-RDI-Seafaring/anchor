Architecture Overview

System Architecture

Anchor UI follows a modern, decoupled architecture with a Next.js frontend and FastAPI backend, connected via REST APIs and WebSocket-like streaming.

│   Next.js UI    │  Port 3000

└────────┬────────┘
         │ HTTP/REST
         │ WebSocket (CopilotKit)
         ▼
┌─────────────────┐
│  Next.js API    │  /api/copilotkit
│     Routes      │
└────────┬────────┘
         │ HttpAgent
         ▼
┌─────────────────┐

    ┌────┴────┐

Data Flow

Chat Message Flow

  1. User types message in frontend
  2. CopilotKit sends to Next.js API route
  3. HttpAgent forwards to FastAPI backend
  4. PydanticAI agent processes message
  5. Agent calls query_knowledge_base tool
  6. Vector search retrieves relevant chunks
  7. LLM generates response with context
  8. Response streamed back through layers
  9. Frontend displays message

Document Upload Flow

  1. User selects file in frontend
  2. File sent directly to FastAPI endpoint
  3. DocumentService saves file
  4. DoclingConverter converts to markdown
  5. MarkdownFormatter creates chunks
  6. EmbeddingsService generates vectors
  7. VectorStore saves chunks and embeddings
  8. Frontend shows success notification

Database Schema

Documents Table

CREATE TABLE documents (
    id SERIAL PRIMARY KEY,
    document_id TEXT UNIQUE NOT NULL,
    filename TEXT NOT NULL,
    file_path TEXT,
    source_type TEXT DEFAULT 'file',
    mime_type TEXT,
    file_size INTEGER,
    chunk_count INTEGER DEFAULT 0,
    status TEXT DEFAULT 'pending',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    metadata JSONB DEFAULT '{}'
);

Chunks Table

CREATE TABLE chunks (
    id SERIAL PRIMARY KEY,
    document_id TEXT NOT NULL,
    chunk_index INTEGER NOT NULL,
    content TEXT NOT NULL,
    embedding vector(3072),
    metadata JSONB DEFAULT '{}',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    UNIQUE(document_id, chunk_index)
);

Indexes

  • HNSW Index: Fast similarity search on embeddings
  • Document ID Index: Quick document lookups

Agent Architecture

PydanticAI Agent

The agent is the core of the AI functionality:

  • Model: Configurable LLM (OpenAI, Azure, Ollama)
  • System Prompt: Defines agent behavior
  • Tools: Extensible tool system
  • State: Conversation and context state

Available Tools

  • query_knowledge_base: Vector similarity search
  • check_db_status: Database health check
  • add_to_conversation: Message history management

Integration Points

CopilotKit Integration

  • Runtime: CopilotRuntime in Next.js API route
  • Agent: HttpAgent connecting to FastAPI
  • Frontend: useCopilotChat hook for UI
  • Streaming: Real-time message streaming

AG-UI Protocol

  • Endpoint: /agent on FastAPI backend
  • Protocol: HTTP-based agent communication
  • Events: State snapshots and tool calls
  • Compatibility: Standard AG-UI client

Security Considerations

Frontend

  • No sensitive data in client code
  • API keys stored server-side only
  • CORS configured for allowed origins

Backend

  • Environment variables for secrets
  • Input validation via Pydantic
  • SQL injection prevention (parameterized queries)
  • File upload validation

Scalability

Current Limitations

  • Single backend instance
  • In-memory active document filter
  • No load balancing

Future Improvements

  • Horizontal scaling with multiple backends
  • Redis for shared state
  • Database connection pooling
  • CDN for static assets

Performance Optimizations

Frontend

  • React memoization for expensive components
  • Code splitting with Next.js
  • Image optimization
  • Lazy loading

Backend

  • Async/await for I/O operations
  • Connection pooling for database
  • HNSW index for fast vector search
  • Caching of embeddings (future)

Error Handling

Frontend

  • Error boundaries for React components
  • Try-catch for async operations
  • User-friendly error messages

Backend

  • HTTP exception handling
  • Structured logging
  • Graceful degradation
